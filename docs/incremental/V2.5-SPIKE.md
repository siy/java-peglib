# V2.5 Spike — GO/NO-GO

**Decision: NO-GO.** v2.5 attacks a cost that does not dominate. The real lever is pivot-selection in `SessionImpl.findBoundaryCandidate`. Defer v2.5 indefinitely.

## Premise summary (SPEC §5.4)

v2.5 keeps the packrat cache alive across edits and remaps it on each edit:
entries entirely before the edit are kept; entries inside `[editOffset, editOffset+oldLen]` are dropped; entries entirely after are key-shifted by `delta` and have their internal `CstNode` spans rewritten. Premise: cache rebuild dominates the per-edit cost; preserving entries outside the edit window will reach < 1 ms.

## Methodology

Throwaway probe `V25SpikeProbe` on the canonical fixture
`peglib-core/.../perf-corpus/large/FactoryClassGenerator.java.txt` (1900 LOC, 101178 chars) parsed with `java25.peg`. JDK 25, no JMH; 5 warmup + 50/30 timed iterations; same `IncrementalParser` as `IncrementalBenchmark`. Edit: insert one space at start of a line (lexically benign trivia, never breaks parseability). Two regimes:

- **Regime A (warm pointer not moved):** initial cursor at offset 60740; sweep edits at fractions 0.05–0.95.
- **Regime B (cursor moved to edit site first):** `session.moveCursor(off)` immediately before each `edit`. Models an editor that tracks caret.

## Cost breakdown (mean ms/op)

| Component | Regime A (no track) | Regime B (track cursor) | Notes |
|---|---:|---:|---|
| Full-parse oracle (inevitable) | 350 | 350 | `oracleParser.parseCst(fixture)` |
| singleCharEdit, same-offset loop | 19.0 (min 12, max 179) | n/a | warm pointer == edit site by construction |
| Sweep mean (30 distinct offsets) | **496** | **39** | the headline number |
| Pivot = `ClassBody` count | **30/30** | 1/30 | ClassBody spans 90878 nodes (≈ entire class) |
| Pivot = small rule (Block/Stmt/Args/...) | 0/30 | 27/30 | means 2.9 – 13.5 ms |
| Fallback to root | 0/50 | 0/30 | never reached root |
| Cache size before/after edit | **0 / 0** | 0 / 0 | `parseRuleAt` allocates a fresh `ParsingContext` each call (`PegEngine.java:384`); v1 has *no* persistent cache |
| Back-reference fallback rule set | **{} (empty)** | {} | `java25.peg` has zero `BackReference` expressions |

The 325 ms/op figure quoted in the release plan corresponds to Regime A
(`IncrementalBenchmark` keeps `warmSession` from `setup()`; per-iteration `edit` does not call `moveCursor`). It is dominated by `parseRuleAt(ClassBody, ...)` which is structurally a full-class reparse (~350 ms), not by cache invalidation.

## Validation findings

**Q1 — Is wholesale cache invalidation the dominant cost?** No. There is no persistent cache to invalidate. `SessionFactory` constructs a single `Parser`, but `Parser.parseRuleAt` (via `PegEngine#parseRuleAt`) calls `ParsingContext.create(...)` on every invocation. The packrat cache lives only for the duration of one `parseRuleAt` call and is never reused across edits. v2.5's "remap" therefore has nothing to remap until persistence is added first — and persistence is the bulk of the implementation work, not the remap.

**Q2 — Does back-reference fallback dominate?** No. `BackReferenceScan.unsafeRules(grammar)` returns `{}` for `java25.peg` (no `$name<e>` capture / `$name` reference syntax in the grammar). 0/30 edits hit `fallbackRules`. This branch is not on the hot path for the IDE-plugin Java target. (It might matter for grammars that *do* use back-references, but those are out of scope for the headline target.)

**Q3 — Can span-rewriting preserve enough entries?** Empirically irrelevant for this fixture, because the *pivot rule itself* covers the bulk of the file. For 30/30 sweep offsets in Regime A, `findBoundaryCandidate` returns `ClassBody`, whose span is essentially `[firstMember, lastMember]` (90878/91100 nodes). Reparsing this rule re-walks the entire interior — every `parseRuleAt` sub-call on every nested rule gets a fresh cache miss. Even with persistent cross-edit caching, every entry inside `[editOffset, editOffset+oldLen]` (≈ 1 char) is fine, but every entry *transitively reachable from `ClassBody`* would still need to validate against the new buffer — and v1's `parseRuleAt` builds them all from scratch. So preservable entries ≈ trivia and rules whose spans lie strictly outside the pivot's span — i.e., `PackageDecl`, `ImportDecl*`, `ModuleDecl?`. That is structurally < 5 % of the rule-instances in this fixture.

## Recommendation rationale

The dominant cost is `findBoundaryCandidate` returning a too-large pivot, not cache rebuild. Concretely: the algorithm walks UP from `enclosingNode` until it engulfs the edit region but never *descends* into the smallest containing child once it has found a containing ancestor. When the warm pointer is far from the edit, the walk overshoots all the way to `ClassBody`. Regime B confirms this — moving cursor to the edit site first drops the sweep mean from **496 ms to 39 ms**, a 12.7× improvement, with no cache work whatsoever.

A real fix is a one-screen change inside `SessionImpl#findBoundaryCandidate`:

1. Locate the smallest node containing `editStart` via `index.smallestContaining(editStart)` (already O(depth)).
2. From there, walk UP only until the span engulfs `[editStart, editEnd]`.

This is essentially Regime B applied automatically per edit, without requiring the editor to call `moveCursor` first. Expected effect: the IDE-plugin's `singleCharEdit` median drops to the **5–15 ms** band on this fixture without any API changes, with zero correctness risk (the pivot is still always a NonTerminal whose span contains the edit; tryIncrementalReparse retains its outward retry).

v2.5 cannot beat that. The minimal pivot's parseRuleAt cost (~3–15 ms for Block/Stmt/Args) IS the parse-itself floor — there is no cache to elide in v1, and even with persistence, the pivot's interior is parsed fresh every time. The remaining headroom toward `< 1 ms` lives in `peglib-core` (per-rule parse cost, the `inlineLocations`/`markResetChildren` work mentioned in SPEC §8 stretch target), not in `peglib-incremental`.

## Alternative levers (NO-GO follow-ups)

Ranked by expected effect ÷ effort. Numbers are projected from the probe.

1. **Edit-anchored pivot selection** (1 day, no API break). Replace warm-pointer walk with `index.smallestContaining(editStart)` + outward walk. Expected `singleCharEdit` median: **5–15 ms** (12–30× speedup). Lands in 0.4.0. Risk: low — pure refactor of one method.
2. **Hybrid: try edit-anchored first, fall back to warm-pointer for adjacent edits** (2 days). Marginal gain over (1) — adjacent edits already inherit a small enclosing node from the previous incremental splice, so warm pointer is fine for them.
3. **Reduce per-rule parse cost in `peglib-core`** (≥ 1 week, separate workstream). Phase-2 `inlineLocations` / `markResetChildren` from the 0.2.2 perf rework spec. Independent of incremental — also speeds up full parse.
4. **Persistent cache + remap (the actual v2.5)** — only worth revisiting after (1) lands. If `singleCharEdit` mean is then ~5 ms and the IDE plugin still reports lag, profile *that* regime to see if cache rebuild has become the new dominant cost. It will not be on this fixture (Regime B already shows pivot-rule parse is the floor), but it might matter for grammars where the typical pivot covers many cacheable sub-rules. Decision to revisit gated on data, not roadmap.

## How to revisit later

Re-run `V25SpikeProbe` (or its successor) under Regime B *after* lever (1) has shipped. Specifically watch:
- mean ms/op on the same fixture (target after lever 1: ≤ 15 ms);
- number of `parseRuleAt` calls per edit (lever 1 = 1 call per edit; v2.5 changes nothing here);
- whether **inside one parseRuleAt call** the packrat cache shows a meaningful hit-rate against an oracle "previous parse" cache (this is the v2.5 hypothesis — probe by enabling persistence on a branch and counting hits).

Re-evaluate v2.5 only if the post-lever-1 mean is still ≥ 50 ms AND the inside-`parseRuleAt` hit-rate would exceed 30 %. Otherwise the engineering cost (cache persistence + span rewrite of cached `CstNode`s, per SPEC §5.4 "subtle correctness note") is not justified.
